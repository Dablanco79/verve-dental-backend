## Purpose

Structured feedback is one of the most valuable outputs of the pilot. This document defines how feedback is collected, categorised, reviewed, and acted upon.

Feedback is not a complaint process — it is the primary mechanism by which the pilot drives platform improvement. Every piece of feedback is valued.

---

## 1. Feedback Categories

All feedback should be assigned to one of the following categories:

| Category | Description | Examples |
|----------|-------------|---------|
| **Bug** | A system behaviour that is incorrect, broken, or behaving unexpectedly | Login fails with correct credentials; adjustment not saving; data showing incorrectly |
| **Improvement** | Something that works but could work better | A filter should persist between sessions; the date picker is hard to use |
| **Training** | A user is confused or uncertain about how to use a feature | Didn't know how to submit a leave request; unsure what a status means |
| **Feature Request** | A capability that does not exist but would be valuable | "Can we see a weekly summary email?"; "Can we bulk approve timesheets?" |
| **Usability** | The interface is confusing, hard to navigate, or inaccessible | Menu labels are unclear; too many clicks to reach a function; text is too small |

> Feedback can belong to more than one category. Select the most relevant primary category and note others in the description.

## 3. Weekly Review Process

### 3.1 Feedback Flow

```
Staff submit feedback to Practice Manager
    ↓
Practice Manager collates feedback into weekly summary
    ↓
Practice Manager submits summary to Pilot Administrator by Friday EOD
    ↓
Daniel Blanco reviews all summaries over the weekend / Monday morning
    ↓
Weekly Pilot Review Call (Monday or Tuesday)
    ├─ Review the week's metrics
    ├─ Discuss feedback themes
    ├─ Assign action items (bugs to log, training to deliver, etc.)
    └─ Communicate resolution updates to Practice Managers
    ↓
Action items tracked through to resolution
    ↓
Resolved items reported back to originating clinic
```

---

### 3.2 Weekly Review Call Agenda

The weekly pilot review call should follow this format:

| Time | Agenda Item |
|------|------------|
| 5 min | Metrics recap (M1–M11 weekly status) |
| 5 min | Incident review (any open or closed incidents) |
| 10 min | Feedback themes from the week |
| 10 min | Action items from last week — status update |
| 5 min | New action items — assign owners and due dates |
| 5 min | Any other business / clinic concerns |

---

### 3.3 Feedback Prioritisation

Not all feedback can be actioned during the pilot. Prioritise as follows:

| Priority | Criteria | Action |
|----------|----------|--------|
| P1 — Immediate | Bug causing data loss, security issue, or operational blocker | Log as incident; fix before next pilot day |
| P2 — This week | Bug impacting multiple users; usability issue causing non-adoption | Investigate and fix within the week |
| P3 — Next sprint | Improvement or usability issue; single-user impact | Log for post-pilot sprint planning |
| P4 — Backlog | Feature request; cosmetic issue; low-impact improvement | Log for product backlog; no immediate action |

---

### 3.4 Closing the Feedback Loop

Every piece of feedback, once actioned, should be acknowledged back to the person who raised it (or their Practice Manager). This is critical for maintaining staff trust in the process.

Example response:

> "Thanks for flagging the issue with the roster date picker last week. We've reviewed it and will address the usability issue in the next update. In the meantime, use the keyboard shortcut [X] as a workaround."

## 5. End-of-Phase Feedback Summary

At the close of each pilot phase, compile a feedback summary including:

- Total feedback items received
- Breakdown by category
- Top 5 themes (positive and negative)
- Issues resolved during the phase
- Issues deferred to post-pilot
- Average user satisfaction score for the phase
- Key quotes from users (anonymised if needed)

This summary forms part of the go/no-go evidence package.
